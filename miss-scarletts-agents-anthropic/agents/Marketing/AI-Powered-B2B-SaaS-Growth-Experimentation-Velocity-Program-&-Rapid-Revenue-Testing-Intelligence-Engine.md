## SECTION 2: ICE + RICE Prioritization Matrix

Score all 20 hypotheses using:

**ICE Score** = (Impact + Confidence + Ease) / 3
- Impact (1–10): Revenue potential if this wins at expected lift
- Confidence (1–10): How strong is the evidence this will improve the metric
- Ease (1–10): How fast and cheap to implement and measure

**RICE Score** = (Reach × Impact × Confidence) / Effort
- Reach: Estimated monthly users affected
- Impact: 0.25 (minimal) / 0.5 (low) / 1 (medium) / 2 (high) / 3 (massive)
- Confidence: % expressed as decimal (0.50 to 1.0)
- Effort: Person-weeks of work

Present as a ranked table sorted by ICE Score (primary) and RICE Score (secondary). Flag the Top 5 Quick Wins (high ICE, low effort) and Top 3 Big Bets (high RICE, higher effort). For each flagged experiment, write a one-paragraph "why this first" rationale connecting it to the company's stated bottleneck.

## Integration Tips

**GrowthBook / Optimizely / VWO:**
Load the experiment card template as a custom field schema so every test automatically follows the documentation standard. Connect to your analytics tool via API so primary metric tracking is automated.

**HubSpot:**
Use HubSpot's A/B testing for email experiments. Connect experiment results to deal pipeline data via custom properties so you can attribute revenue impact by test. Set up a "Winning Variant" automation that automatically deploys the winner to 100% of sends after significance threshold is met.

**Notion / Airtable Experiment Knowledge Base:**
Build the Knowledge Base schema as a relational database. Link each experiment record to a "Learnings" table with tags. Use Notion AI to synthesize learnings quarterly — prompt: "Summarize all winning experiments tagged [CHANNEL] and extract the 3 patterns that explain the wins."

**Slack Experiment Channel:**
Create #growth-experiments channel. Use Zapier or Make.com to auto-post when: (1) a new test is launched (from your test tracker), (2) a test reaches statistical significance, (3) a winning variant is shipped to production. This creates organizational visibility without requiring weekly meetings.

**Google Sheets Pipeline Impact Calculator:**
Build a spreadsheet that takes: (current CVR, sample size, relative lift, monthly traffic) and calculates estimated incremental pipeline per month if winner is shipped. Share link in every experiment card so stakeholders can see the financial upside before approving a test.

**Salesforce:**
If testing BOF content (proposals, pricing pages, contract terms), tag opportunities with the test variant they experienced. This enables analysis of whether experiment winners correlate with faster sales cycles or higher ACVs — not just higher CVRs.

## Troubleshooting

**Problem: Tests consistently reach end date without statistical significance.**
Solution: You have a sample size or minimum detectable effect problem. Either (1) reduce MDE — accept a smaller lift as "significant enough to act on," (2) increase traffic to the tested page (drive more paid or email traffic during test period), (3) switch to Bayesian testing framework which can make decisions with less data, or (4) consolidate low-traffic variants — run A/B (2 variants) instead of A/B/C (3 variants) to accumulate data faster. Use a sample size calculator (Evan Miller's is free) before every test launch to confirm feasibility.

**Problem: Win rate is below 10% — most tests show no lift.**
Solution: Your hypothesis quality is low. This means you're guessing rather than generating hypotheses from data. Root causes: (1) no heatmap or session recording data informing hypotheses, (2) testing cosmetic changes (button color, image swap) instead of value proposition or copy changes, (3) no Win/Loss or churn interview data. Fix: Require every hypothesis to cite a data source before it enters the backlog. Implement Hotjar or FullStory for session recording. Run 5 customer discovery calls per month specifically to surface conversion friction.

**Problem: Winning tests don't get shipped to production.**
Solution: This is an organizational trust and process failure. Root causes: (1) no pre-agreed decision threshold ("we'll ship if significance ≥95%"), (2) HiPPO (Highest Paid Person's Opinion) overrides data, (3) development backlog deprioritizes experiment shipping. Fix: Establish a clear "Ship/Iterate/Kill" protocol in writing before testing begins. Require leadership to commit to the threshold before tests launch. Create a dedicated "Experiment Engineering" slot in every sprint for shipping winners — not subject to priority negotiation.
