## Integration Tips

- **Optimizely / VWO / Google Optimize alternatives**: Pipe experiment variant assignments into HubSpot via custom contact field — this links test variants to downstream CRM pipeline data, not just web conversions, enabling true revenue-level attribution
- **GA4 + BigQuery**: Export experiment data to BigQuery for custom Bayesian analysis scripts — use Google's free Colab notebooks with PyMC or Stan libraries for full posterior distributions on conversion rates
- **HubSpot Workflows**: Create automated workflow that tags every contact who converted during an active experiment with the variant they saw — enables post-hoc cohort analysis months later when deals close
- **Salesforce Campaigns**: Create one Salesforce Campaign per experiment variant — this lets sales managers filter pipeline by experiment cohort and validates whether conversion rate improvements hold through the full funnel
- **Slack Integration**: Set up automated Slack alerts when any running experiment crosses 95% probability threshold (using Optimizely webhooks or custom GA4 → Zapier → Slack pipeline) — reduces delay between statistical significance and business decision
- **Notion / Confluence Learning Repository**: Templatize the Learning Repository entry (Test ID, hypothesis, result, mechanism, next experiment) and automate creation via Zapier when experiment is marked "complete" in your testing tool

## Troubleshooting

**Issue: Test runs for 8+ weeks without reaching statistical significance**
Solution: You're likely underpowered. Either (1) increase MDE — accept a smaller but still meaningful effect, (2) consolidate traffic to a single high-value variant instead of multi-variate, (3) switch to Bayesian analysis which provides interpretable results at any sample size rather than requiring fixed sample sizes, or (4) use Sequential Testing / Group Sequential Methods that allow early stopping with controlled error rates. If the page genuinely has < 200 conversions/month, run qualitative research (user testing, session replay) instead — statistical testing requires sufficient traffic to be meaningful.

**Issue: Test shows a winner but revenue metrics don't improve after rollout**
Solution: Your primary KPI was measuring the wrong thing. If demo requests increased but pipeline didn't, you attracted lower-quality visitors. Add guardrail metrics from the start: track MQL quality score, demo show rate, and demo-to-opportunity rate as secondary metrics — a "win" that degrades downstream metrics is a false win. Retrospectively segment results by ICP firmographic cohort (company size, industry, seniority) — the "winner" may only work for a subset of visitors.

**Issue: Multiple tests running simultaneously produce conflicting or confounded results**
Solution: Implement a traffic partitioning system: never run more than one test on the same URL or user segment simultaneously without isolation. Use a testing calendar to ensure maximum 1 active test per major touchpoint. If you must run concurrent tests, use a holdout group (10-15% of traffic unexposed to any test) as your true baseline. For email sequences, use list segmentation to ensure contacts only enter one experiment at a time — HubSpot's A/B send feature handles this automatically but custom workflows require manual audience isolation.
