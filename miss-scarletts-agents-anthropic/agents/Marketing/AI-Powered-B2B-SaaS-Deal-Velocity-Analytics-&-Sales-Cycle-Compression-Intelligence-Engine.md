## Troubleshooting

**"My CRM stage-entry timestamps are unreliable — AEs backdating stages"**
Fix: Stop measuring time-in-stage from stage date; instead use Activity Date of the event that should signal stage entry (e.g., first Gong call in Discovery = Discovery start). Pull from Gong/Chorus API rather than CRM stage date. Alternatively, implement a required field ("Stage Progression Verified by Manager") that creates an audit trail and reduces backdating behavior.

**"The same velocity killer shows up in every stage — it's just slow AEs"**
Fix: Break VPI analysis by AE cohort (top 25% vs. bottom 25% velocity). If fast AEs exist, the issue is process and enablement, not team capability. Document exactly what fast AEs do differently in each stage (call recordings, email templates, content used) and systematize it into the playbook. If ALL AEs are slow, the bottleneck is structural — product/market fit, ICP clarity, or pipeline quality, not velocity tactics.

**"We implemented all the interventions but cycle stayed the same"**
Fix: Audit asset adoption first — if AEs aren't using the velocity accelerators, cycle won't move. Check Highspot/Seismic analytics. If adoption is >60% but velocity didn't move, the intervention matched the wrong killer. Re-run Module 3 diagnosis — sometimes the visible symptom (deal stalling at Proposal) has a hidden root cause (wrong ICP was qualified in, so economic buyer never has urgency). Velocity compression requires correct ICP qualification upstream.
