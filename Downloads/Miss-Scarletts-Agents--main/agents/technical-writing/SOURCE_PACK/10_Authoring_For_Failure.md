# Authoring for Failure

Documentation should help when things go wrong, not only when things go right.

## Core facts

- Error handling guidance recommends reporting failures instead of failing silently.
- The most useful failure message usually identifies the cause, the invalid input, or the missing requirement.

## Practical guidance

- Write docs and messages that help a reader recover.
- Explain what the reader should do next.
- Keep long explanations accessible through links or secondary sections.

## Common mistakes

- Only documenting the happy path.
- Leaving the reader without a recovery action.
- Turning the failure path into a wall of text.
