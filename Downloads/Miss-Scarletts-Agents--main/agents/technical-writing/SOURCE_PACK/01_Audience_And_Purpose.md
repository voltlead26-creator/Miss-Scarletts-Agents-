# Audience and Purpose

The first question in documentation is who the reader is and what they are trying to do.

## Core facts

- Audience determines vocabulary, depth, and examples.
- A document for a new user should not read like a reference for an expert.
- The purpose of the document should be obvious from the first section.

## Practical guidance

- Write for the reader's current task, not for the system's internal structure.
- State what the document will help them do.
- Avoid assuming background knowledge that the reader may not have.

## Common mistakes

- Writing around internal implementation instead of the user's need.
- Using the same explanation style for all audiences.
- Hiding the purpose in a long introduction.
